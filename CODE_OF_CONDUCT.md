# Code of Conduct

Orkestra is a small open-source project. The contributors are humans, the maintainers are humans, and the bug reporters are humans. This document is how we keep it that way.

---

## 🤝 The Short Version

> **Be the kind of person you'd want to work with at 2 AM debugging a production outage.**

Patient. Direct. Willing to be wrong. Willing to help.

If you can hold those four things, you'll never need the rest of this document.

---

## ✅ What We Value

| Principle | What it looks like in practice |
|-----------|--------------------------------|
| **Curiosity over certainty** | "I wonder why this fails" beats "this is broken" |
| **Evidence over opinion** | Logs, repros, and stack traces beat vibes |
| **Critique the code, not the coder** | "This function leaks a goroutine" — not "you don't understand Go" |
| **Assume good intent** | Most weird PRs come from missing context, not malice |
| **Boring is fine** | A boring fix that ships beats a brilliant one that doesn't |
| **Time has weight** | Reviewers, maintainers, and reporters all donate hours. Respect that |

---

## 🚫 What Doesn't Belong Here

These get you removed without negotiation:

- Personal attacks, slurs, or harassment — public or private
- Doxxing or sharing someone's info without consent
- Sexualized language or unwelcome advances
- Sustained disruption of issues, PRs, or discussions
- Retaliation against anyone who reports a problem in good faith

The list isn't exhaustive. If you're wondering whether something crosses the line, it probably does.

---

## 🌐 Where This Applies

Anywhere the project lives:

```
GitHub Issues  →  Pull Requests  →  Discussions  →  Commit messages
                          ↓
                    Anyone speaking
                  on behalf of Orkestra
```

This includes private DMs that spill from a public Orkestra context.

## 🚨 Reporting

If something feels wrong, tell the maintainer:

📧 **vijayanand431@gmail.com**

Include what happened, where, when, and (if possible) links. Your report stays confidential — only the maintainer reads it. You will get a reply within 7 days.

If the maintainer is the problem, post in [GitHub Discussions](https://github.com/Vijay431/Orkestra/discussions) and tag it `coc`. Other contributors will pick it up.

---

## ⚖️ How Enforcement Works

There's no committee. The maintainer reads the report, talks to the people involved, and picks a response from this ladder:

```
1. Private nudge       → "Hey, this came across as X. Can you adjust?"
2. Public correction   → A note on the thread, expected to be acknowledged
3. Temporary timeout   → 7-day ban from issues / PRs / discussions
4. Permanent ban       → Removal from the project, no return path
```

Severity decides where you land. A first-time misstep usually gets a nudge. Targeted harassment lands at step 4.

## 🪞 Maintainers Aren't Exempt

Everything here applies to me too. If I'm rude in a code review, ghost a PR for two months, or shut down a legitimate concern — call it out. Public or private, your choice.
